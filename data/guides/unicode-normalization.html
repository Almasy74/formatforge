What Is Unicode Normalization?

Unicode normalization is the process of converting Unicode text into a single, consistent byte representation so that visually identical characters always produce the same byte sequence. This matters because Unicode allows many characters — like accented letters, ligatures, and symbols — to be encoded in more than one way. Without normalization, two strings that look identical on screen can fail equality checks, break database lookups, and produce inconsistent search results.

The Unicode Standard defines four normalization forms: NFC, NFD, NFKC, and NFKD. The two most important for everyday development are NFC (composed) and NFD (decomposed). Understanding the difference between them — and knowing when each causes problems — is the key to avoiding subtle encoding bugs in your applications.

NFC vs NFD: The Core Difference

NFC (Canonical Decomposition, followed by Canonical Composition) combines base characters and their combining marks into single precomposed code points wherever possible. NFD (Canonical Decomposition) does the opposite — it splits precomposed characters into their base character plus separate combining marks.

Property NFC (Composed) NFD (Decomposed)
How it stores é U+00E9 (1 code point) U+0065 U+0301 (2 code points)
Byte length (UTF-8) 2 bytes 3 bytes
Visual appearance é é (identical)
String equality NFC ≠ NFD without normalization
Best for Storage, databases, APIs, search indexing Text analysis, character-level processing
Default in W3C, HTML, most databases macOS HFS+ filenames

All Four Unicode Normalization Forms

Form Full Name What It Does Use Case

NFD Canonical Decomposition Fully decomposes into base + combining marks Character analysis, sorting

When to Use NFC

NFC is the default recommendation for almost all production scenarios. The W3C recommends NFC for all web content. Use NFC when you need:

  • Database storage — consistent byte representation ensures WHERE name = 'café' matches every time
  • Search indexing — normalized terms prevent missed matches when users type accented queries
  • URL slug generation — NFC ensures consistent transliteration of diacritics before slug creation
  • API payloads — consistent encoding prevents checksum and signature mismatches
  • User authentication — username and password comparison must normalize first to avoid false rejections

When NFD Causes Problems

NFD is rarely the right default, but it appears more often than you might expect — macOS's HFS+ filesystem stores filenames in NFD form, and some text editors preserve decomposed input. Here are common failure scenarios:

  • Regex doesn't match — the pattern /[a-z]/ matches the base e in NFD but not the combining accent U+0301, causing partial matches
  • String length is wrong — "é".length returns 1 in NFC but 2 in NFD, breaking character limits and truncation logic
  • Database lookups fail — if one row was inserted as NFC and another as NFD, a SELECT DISTINCT query returns "duplicates" that look identical
  • File deduplication breaks — two files with the same NFD name on macOS and NFC name on Windows appear as different files in version control
  • Text analysis gives inflated character counts when combining marks are counted separately

Code Examples

JavaScript — String.prototype.normalize()

// The "Identity Paradox" — looks the same, fails equality
const nfc = "\u00E9";           // é as precomposed (NFC)
const nfd = "e\u0301";          // e + combining acute accent (NFD)

console.log(nfc === nfd);       // false — different byte sequences!
console.log(nfc.length);        // 1
console.log(nfd.length);        // 2

// Fix: normalize both to the same form
console.log(nfc === nfd.normalize("NFC"));  // true ✓

// Practical pattern: always normalize user input
function sanitizeInput(text) {
    return text.normalize("NFC").trim();
}

// Slug generation: normalize before transliteration
function toSlug(title) {
    return title
        .normalize("NFC")
        .toLowerCase()
        .replace(/[\u0300-\u036f]/g, "")  // strip combining marks
        .replace(/[^a-z0-9]+/g, "-")
        .replace(/^-|-$/g, "");
}

Python — unicodedata.normalize()

import unicodedata

nfc = "\u00E9"                    # é precomposed
nfd = "e\u0301"                   # e + combining acute

print(nfc == nfd)                 # False
print(len(nfc), len(nfd))         # 1 2

# Normalize to NFC
normalized = unicodedata.normalize("NFC", nfd)
print(nfc == normalized)          # True ✓

# Remove all diacritics (useful for search indexing)
def strip_accents(text):
    nfkd = unicodedata.normalize("NFKD", text)
    return "".join(c for c in nfkd if not unicodedata.combining(c))

print(strip_accents("café résumé"))  # "cafe resume"

# Database normalization pattern
def db_normalize(value):
    """Normalize text before database INSERT or comparison."""
    if isinstance(value, str):
        return unicodedata.normalize("NFC", value).strip()
    return value

Frequently Asked Questions

Should I always use NFC?

For storage, databases, web content, and APIs: yes, NFC is the W3C and Unicode Consortium's default recommendation. Use NFD only when you specifically need character-level decomposition (e.g., removing accents for search indexing).

Does JavaScript normalize strings automatically?

No. JavaScript strings preserve whatever form they were created in. You must explicitly call .normalize("NFC") to convert. This applies to user input, fetch responses, and file content alike.

What about NFKC and NFKD — when do I need those?

NFKC and NFKD also handle "compatibility" equivalences — for example, converting the ligature ﬁ to fi, or the superscript ² to 2. Use NFKC for search indexing or username comparison where you want maximum compatibility. Avoid it for display text, since it can change visual meaning.

How do I detect normalization issues in existing data?

Paste your text into FormatForge's Text Analyzer to inspect character counts and detect hidden combining marks or zero-width characters. If a string's character count differs from its visual length, you likely have normalization inconsistencies.

Do databases normalize Unicode automatically?

Most do not. PostgreSQL, MySQL, and SQLite store strings as-is. You must normalize in your application layer before INSERT and before comparison queries. Some collations treat NFC and NFD as equal for sorting, but byte-level comparison (used in WHERE clauses and unique constraints) does not.
